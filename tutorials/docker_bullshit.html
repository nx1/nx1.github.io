nx1.info | An Investigation into Anomalous Container Persistence on Embedded GPU Platforms

The management of complex microservices via Docker Compose typically provides a deterministic environment for Edge AI deployments. However, edge-case failure modes occasionally manifest as "persistent phantom processes"—containers that resist standard termination signals and survive system reboots. In this report, I provide a forensic analysis of a failure mode encountered during the deployment of a multi-container machine learning pipeline (Smartbench) on an NVIDIA Jetson Orin NX. I begin by describing the observed symptoms and the failure of standard diagnostic procedures. I show how a discrepancy between the Docker daemon's state and the physical filesystem led to a paradoxical "No such container" error. The report concludes with a definitive resolution involving the manual reconciliation of the Docker root directory.

I. The Manifestation of Synchronisation Failure

The deployment architecture utilized a run_smartbench.sh watchdog script to orchestrate the docker compose up sequence. The design intent of this script was to initiate a graceful shutdown of all services upon the detection of a critical failure in any constituent container. Initial execution of the deployment script resulted in an immediate transition to the shutdown sequence. Examination of the daemon output revealed a critical exception: 
Error response from daemon: No such container: 042fd7bc7537...
This indicated a loss of synchronisation between the Docker Compose state and the underlying Docker Engine, where the former attempted to manage a container ID that the latter could no longer reference.

II. Diagnostic Isolation and Dependency Verification

Initial efforts focused on the isolation of individual containerized services to verify the integrity of the application layer. By decoupling the services from the Compose orchestration and executing them interactively via docker run, I was able to identify a primary failure mode. The GUI and Data Manager services were found to be terminating prematurely due to a missing runtime dependency—a custom logger.py script. While the provision of a volume mount (-v /path/to/logger.py:/app/logger.py) resolved the application-level instability, the original docker compose up error persisted. This confirmed that the root cause was localized within the Docker Engine's state management rather than the application code.

III. Evaluation of Standard Recovery Procedures

Standard procedures for reconciling the Docker daemon state typically involve the pruning of orphan resources and force-recreating the stack. The following gauntlet was executed: 
docker compose down --remove-orphans
docker network prune -f
docker compose up -d --force-recreate
Despite these measures, the daemon continued to report the non-existence of container 042fd7bc7537. This persistence across multiple prune operations suggested a more profound corruption of the metadata.

IV. The Paradox of the "Ghost" Container

A global audit of the container list via docker ps -a revealed the existence of the problematic container ID, showing a status of Exited (143) 6 weeks ago. However, attempting to remove the container directly yielded a paradoxical result: 
matoha@orin-nx-1:~$ docker rm -f 042fd7bc7537
Error response from daemon: No such container: 042fd7bc7537
The container was simultaneously present in the process list and absent from the daemon's internal registry. Further attempts to flush the state by restarting the containerd and docker services proved unsuccessful, even after the termination of any lingering containerd-shim processes.

V. Forensic Analysis of the Filesystem

When a container survives a full service restart but cannot be removed via the CLI, it implies that the metadata database is corrupted. The physical folder representing the container exists on the drive, but the JSON state file inside is either missing or unreadable. To resolve this, I navigated to the default Docker root (/var/lib/docker/containers). Surprisingly, the directory appeared to be empty: 
root@orin-nx-1:/var/lib/docker/containers# ls -l
total 0
This discrepancy led to the realization that the storage configuration had been modified. On specialized hardware like the Jetson Orin NX, NVMe SSDs are often utilized to mitigate the wear of internal eMMC storage. An audit of the system configuration confirmed this hypothesis: 
root@orin-nx-1:/var/lib/docker/containers# docker info | grep "Docker Root Dir"
 Docker Root Dir: /ssd/docker

VI. Conclusion and Resolution

By identifying the true location of the Docker root, the resolution became surgical. The process involved the following steps: 1. Termination of the Docker service to unlock the filesystem. 2. Removal of the corrupted metadata folder on the SSD: rm -rf /ssd/docker/containers/042fd7bc753741e23065557633643464bb51c3ebf78baa03b9a8f17ed568ea3a/ 3. Restoration of the Docker service. Following the manual deletion of the corrupted folder, docker ps -a returned a clean state. The run_smartbench.sh script subsequently initialized all 13 containers flawlessly. This investigation highlights the importance of verifying the underlying storage architecture on non-standard platforms and the utility of manual metadata reconciliation when the Docker daemon becomes decoupled from the filesystem state.
